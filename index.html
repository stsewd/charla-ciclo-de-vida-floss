Al principio todo era oscuridad. Estamos frente al computador, observando nuestro cursor parpadear, nuestra terminal esperando que escribamos un comando. Un proyecto empieza así, empieza con una necesidad, o simplemente con una idea. Algo nuevo o divertido que queremos probar.

$ mkdir awesome-project


          $ cd awesome-project

Empieza con un directorio vacío, con un archivo, con una línea de código. O si prefieren la interfaz gráfica, empieza con un par de clicks.

Codeando hasta que sea perfecto

Pero nunca lo será.

Pero está lo suficientemente cerca para que sea de utilidad a alguien más.

Nos quedamos codeando todo el día, toda la noche, furiosamente golpeando el teclado para darle forma a nuestra idea. Funciona, pero aún no es perfecto, nunca lo será. Pero está lo suficientemente cerca para que sea de utilidad a alguien más, o simplemente para que otras personas vean lo terrible que programamos. Así que decidimos compartirlo.

$ git init

  • ¿Quién? ¿Qué? ¿Cuándo?
  • Mantener un registro de los cambios en el código.
  • Colaboración con otros usuarios.
  • Dar crédito a los autores cuando incluyas sus cambios.
Al menos que estemos en los 90, debemos usar un sistema de control de versiones. Git es una opción popular. Si no están familiarizados con Git, hay muchos recursos en línea. Un sistema de control de versiones nos permite mantener un registro de los cambios en el código, ayudándonos a responder a las preguntas: ¿Quién? ¿Qué? ¿Cuándo? Además nos facilita la colaboración con otros usuarios, mediante el uso de ramas y parches, y sobre todo podemos dar crédito a los autores cuando incluyas sus cambios.

$ git push

Compartiendo tu código

El siguiente paso es compartir tu código, para ello necesitamos un sitio donde hostearlo. Un lugar que pueda ser accedido fácilmente por el Internet. GitHub o GitLab son una opción popular.

$ git push

¡Pero espera!

Pero espera, Pero simplemente subir tu código no es suficiente para que otras personas lo usen. Necesitamos:documentar nuestro proyecto, y agregarle una licencia.

Documentación

README.md

docs/

  • Sphinx
  • MkDocs
  • Docusaurus
Documentar nuestro proyecto puede ser una tarea aburrida, pero un simple archivo README explicando lo que hace y cómo usarlo es suficiente para empezar. O si nuestro proyecto es mucho más grande, podríamos considerar algo más estructurado, usando herramientas para crear documentación como Sphinx, MkDocs o Docusaurus. Y recuerda, que la documentación no necesariamente está siempre en archivos separados, puede estar en el mismo código, en forma de docstrings o comentarios.

¿Qué incluir en la documentación?

  • Descripción del proyecto
  • Instalación
  • Uso y ejemplos
  • Cómo contribuir

Licencia

Reglas y acuerdos que las personas que usan o modifican tu proyecto deben seguir.

Una licencia es esencial para que tu proyecto pueda ser usado por otras personas. Especifica ciertas reglas y acuerdos que las personas que usan o modifican tu proyecto deben seguir.

¿Qué licencia usar?

LICENSE

  • GPL, AGPL, LGPL
  • MIT
  • Apache
  • Creative Commons
Qué licencia usar depende de tu proyecto, tus objetivos y de tus necesidades. Unas opciones populares son la familia de las licencias GPL, o una licencia más permisiva como MIT o Apache. Y si tienes un proyecto que no sea código, puedes usar una licencia Creative Commons, que es una licencia más adecuada para este tipo de proyectos.

¿Qué licencia usar?

LICENSE

  • https://docs.codeberg.org/getting-started/licensing/
  • https://choosealicense.com/
Para escoger tu licencia puedes usar la guía de Codeberg, o la web choosealicense.com. Y usar una licencia en tu proyecto es tan fácil como crear un archivo llamado LICENSE en la raíz de tu proyecto, o si te quieres poner más extricto, puedes poner tu licencia en cada archivo.

El primer pull request

$ git merge feature

Probablemente tú vas a ser el autor del primer pull request. Pero conforme tu proyecto crezca, más usuarios experimentados van a querer contribuir. Algunas van a adaptar tu proyecto para sus necesidades, Van a querer enviarte esas mejoras. Las mismas personas que reportaron issues van a querer contribuir con soluciones.

No estás obligado a hacer nada de esto

  • Responder preguntas
  • Solucionar problemas
  • Tomar decisiones
  • Code review
  • Aceptar pull requests
Conforme tu proyecto crece, tus usuarios van a esperar que estés disponible para responder sus preguntas, solucionar problemas, hacer code review, y aceptar pull requests, etc. Puede consumir mucho tiempo, y puede llegar a ser frustrante. Mientras sigas disfrutando de tu proyecto, está bien, pero si llega a ser una carga, recuerda que no estás obligado a mantenerlo. Nadie te está pagando por hacerlo, y no hay nada de malo en dejarlo morir.
